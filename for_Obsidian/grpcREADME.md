# UDP-gRPC Proxy Project

Этот проект реализует систему для удаленного управления устройствами, подключенными к COM-порту, с использованием UDP для клиентского интерфейса и gRPC для связи между прокси и сервером.

## Архитектура и Принцип работы

Система состоит из следующих основных компонентов:

1.  **UDP Клиент (`client/udp_client.py`):**
    *   Предоставляет пользователю интерфейс командной строки для отправки команд.
    *   Формирует команды в виде байтовых последовательностей (часто в HEX-формате).
    *   Добавляет CRC к данным для проверки целостности (используя `shared/crc_utils.py`).
    *   Отправляет UDP-пакеты с командами на известный адрес и порт прокси-сервера.
    *   Ожидает и отображает ответ, полученный от прокси по UDP.

2.  **UDP-to-gRPC Прокси (`proxy/udp_to_grpc_proxy.py`):**
    *   **Слушает UDP:** Принимает UDP-пакеты от одного или нескольких клиентов на своем порту (например, 9999).
    *   **Проверяет CRC:** Проверяет и удаляет CRC из полученных данных, используя `shared/crc_utils.py`.
    *   **Парсит команду:** Анализирует байтовую последовательность команды, определяя целевой блок (БУ, БЗ, БФ), тип устройства и команду.
    *   **Маршрутизирует запрос:** Направляет запрос к соответствующему gRPC серверу в зависимости от целевого блока.
    *   **Преобразует в gRPC:** Создает структурированный gRPC запрос на основе разобранной команды.
    *   **Вызывает gRPC Сервер:** Отправляет запрос на соответствующий сервер блока и получает ответ.
    *   **Преобразует ответ в UDP:** Конвертирует данные из ответа gRPC обратно в байтовую последовательность.
    *   **Добавляет CRC:** Добавляет CRC к байтам ответа.
    *   **Отправляет ответ Клиенту:** Отправляет UDP-пакет с ответом обратно исходному клиенту.

3.  **gRPC Серверы:**
    *   **Сервер блока БУ** (`server/grpc_com_server_BU.py`) - обслуживает устройства блока БУ (блок управления).
    *   **Сервер блока БЗ** (`server/grpc_com_serverBZ.py`) - обслуживает устройства блока БЗ (блок защиты).
    *   **Сервер блока БФ** (`server/grpc_com_serverBF.py`) - обслуживает устройства блока БФ (блок формирования).
    
    Каждый сервер:
    *   **Реализует gRPC сервис:** Предоставляет реализацию методов, определенных в `device_control.proto`.
    *   **Принимает запросы:** Слушает gRPC запросы от прокси на своем порту.
    *   **Взаимодействует с устройствами:** Управляет устройствами своего блока через COM-порт.
    *   **Формирует ответы:** Создает и отправляет структурированные gRPC ответы обратно прокси.
    *   **Обрабатывает команды ШСКМ:** Все серверы (БУ, БЗ, БФ) обрабатывают команды для блока ШСКМ (0x0), так как отдельного сервера ШСКМ не предусмотрено.

**Таким образом, система обеспечивает масштабируемую архитектуру, где прокси выступает как единая точка доступа для клиентов, а серверы обслуживают различные физические блоки. Это позволяет скрыть сложность gRPC и работы с оборудованием от конечного клиента.**

## Структура проекта

- `client/`: Содержит код UDP клиента (`udp_client.py`).
- `proxy/`: Содержит код UDP-to-gRPC прокси (`udp_to_grpc_proxy.py`).
- `server/`: Содержит код gRPC серверов для разных блоков:
    - `grpc_com_server_BU.py`: Сервер блока БУ.
    - `grpc_com_serverBZ.py`: Сервер блока БЗ.
    - `grpc_com_serverBF.py`: Сервер блока БФ.
- `shared/`: Содержит общие утилиты и сгенерированные файлы:
    - `crc_utils.py`: Функции для расчета и проверки CRC.
    - `device_control_pb2.py`: Сгенерированный Python код для сообщений protobuf.
    - `device_control_pb2_grpc.py`: Сгенерированный Python код для gRPC клиента и сервера.
- `device_control.proto`: Файл определения сервиса и сообщений Protocol Buffers для gRPC (в корне проекта).
- `requirements.txt`: Список зависимостей Python.
- `run_local_test.ps1`: PowerShell скрипт для автоматического запуска компонентов в Windows.
- `README.md`: Этот файл.
- `testLocalREADME.md`: Инструкция по запуску локального тестирования в Windows.

## Установка зависимостей

Проект использует инструменты `uv` для быстрой установки пакетов и `ruff` для проверки качества кода.

### Установка uv и ruff (если они еще не установлены)

```bash
# Установка pipx (если не установлен)
python -m pip install pipx
# Установка uv
pipx install uv
# Установка ruff
pipx install ruff
```

### Создание виртуального окружения с помощью uv

```bash
# Создание виртуального окружения в директории .venv312uv
uv venv .venv312uv

# Активация виртуального окружения
# Windows (PowerShell)
.\.venv312uv\Scripts\Activate.ps1
# Windows (CMD)
.\.venv312uv\Scripts\activate.bat
# Linux/macOS
source .venv312uv/bin/activate
```

### Установка зависимостей с помощью uv

```bash
# Установка всех зависимостей из requirements.txt
uv pip install -r requirements.txt
```

### Автоматическая проверка кода с помощью ruff

```bash
# Проверка всего проекта
ruff check . --select=E,F,W --ignore=E501 --exclude=__pycache__,\.venv,\.git

# Автоматическое исправление простых проблем
ruff check . --select=E,F,W --ignore=E501 --exclude=__pycache__,\.venv,\.git --fix
```

### Генерация protobuf файлов

Если вы вносили изменения в `device_control.proto` (в корне проекта), пересоберите protobuf-файлы:

```bash
# Выполните из корневой директории проекта
python -m grpc_tools.protoc -I. --python_out=./shared --grpc_python_out=./shared ./device_control.proto
```

Что делает эта команда:
- `-I.`: Искать импортируемые `.proto` файлы (если они есть) в текущей директории (корне проекта).
- `--python_out=./shared`: Поместить сгенерированный `device_control_pb2.py` в папку `shared`.
- `--grpc_python_out=./shared`: Поместить сгенерированный `device_control_pb2_grpc.py` в папку `shared`.
- `./device_control.proto`: Указывает исходный `.proto` файл в текущей (корневой) директории.

После выполнения команды в папке `./shared` должны появиться (или обновиться) файлы:
- `device_control_pb2.py`
- `device_control_pb2_grpc.py`

## Запуск с помощью скрипта в Windows

Для удобного тестирования в Windows можно использовать автоматический скрипт `run_local_test.ps1`:

```powershell
# Запуск в интерактивном режиме (с запросом настроек)
.\run_local_test.ps1

# Запуск без интерактивного режима (с настройками по умолчанию)
.\run_local_test.ps1 -NoInteractive

# Запуск без проверки кода линтером ruff
.\run_local_test.ps1 -SkipLinting

# Комбинирование параметров
.\run_local_test.ps1 -NoInteractive -SkipLinting
```

Скрипт автоматически:
1. Устанавливает uv и ruff (если они отсутствуют)
2. Создает виртуальное окружение
3. Устанавливает все зависимости
4. Проверяет код с помощью ruff (если не указан параметр -SkipLinting)
5. Запускает все три сервера (БУ, БЗ, БФ)
6. Запускает UDP-to-gRPC прокси
7. Запускает UDP клиент в отдельном окне

Подробная инструкция по использованию скрипта находится в файле `testLocalREADME.md`.

## Ручной запуск приложений

Откройте необходимое количество терминалов или консолей (для запуска одного или нескольких серверов блоков и прокси). **В каждом из них** сначала активируйте виртуальное окружение `.venv312uv`, как описано выше. Затем выполните следующие команды:

1.  **Запуск gRPC серверов для разных блоков:**
    (В отдельных терминалах, с активированным `.venv312uv`)
    
    * **Сервер блока БУ:**
      ```bash
      python server/grpc_com_server_BU.py --com-port /dev/cu.usbserial-21430 --port 50051
      python server/grpc_com_server_BU.py --com-port COM3 --port 50051
      # Режим отладки без COM-порта
      python server/grpc_com_server_BU.py --no-com --debug --port 50051
      ```
    
    * **Сервер блока БЗ:**
      ```bash
      python server/grpc_com_serverBZ.py --com-port /dev/cu.usbserial-21431 --port 50052
      python server/grpc_com_serverBZ.py --com-port COM4 --port 50052
      # Режим отладки без COM-порта
      python server/grpc_com_serverBZ.py --no-com --debug --port 50052
      ```
    
    * **Сервер блока БФ:**
      ```bash
      python server/grpc_com_serverBF.py --com-port /dev/cu.usbserial-21432 --port 50053
      python server/grpc_com_serverBF.py --com-port COM5 --port 50053
      # Режим отладки без COM-порта
      python server/grpc_com_serverBF.py --no-com --debug --port 50053
      ```
    
    Каждый сервер запускается на своем порту и работает со своим COM-портом.

2.  **Запуск UDP-to-gRPC Прокси:**
    (В отдельном терминале, с активированным `.venv312uv`)
    ```bash
    python proxy/udp_to_grpc_proxy.py
    ```
    Прокси запустится, будет слушать UDP на адресе `127.0.0.1` и порту `9999` (по умолчанию) и перенаправлять запросы на соответствующие gRPC серверы блоков.

3.  **Запуск UDP Клиента:**
    (В отдельном терминале, с активированным `.venv312uv`)
    ```bash
    python client/udp_client.py
    ```
    Клиент запустится и предложит меню для отправки UDP данных на прокси.

## Примеры команд клиента

После запуска `client/udp_client.py` вам будет предложено меню. Вот примеры команд в HEX формате для различных блоков:

### Команды для ШСК_М
*   **ШСК_М Старт:**
    ```
    00 00 01
    ```
*   **ШСК_М Стоп:**
    ```
    00 00 00
    ```

**Примечание:** Команды для ШСК_М (0x0) обрабатываются всеми тремя серверами (БУ, БЗ, БФ). Это позволяет системе работать без создания отдельного сервера для ШСК_М.

### Команды для Блока БУ (Блок 0x1)
*   **Чтение серийного номера БУ:**
    ```
    01 01 00
    ```
*   **Чтение состояния БУ:**
    ```
    01 10 00
    ```
*   **Чтение/установка ослабления БУ:**
    ```
    01 11 00    # Чтение
    01 11 01 XX # Установка, где XX - значение
    ```
*   **Установка ВЧ переключателя S1 БУ:**
    ```
    01 41 01 01 # Включить
    01 41 01 00 # Выключить
    ```
*   **Чтение датчика температуры 1 БУ:**
    ```
    01 51 00
    ```
*   **Чтение индикатора напряжения 1 БУ:**
    ```
    01 61 00
    ```

### Команды для Блока БЗ (Блок 0x2)
*   **Чтение серийного номера БЗ:**
    ```
    02 01 00
    ```
*   **Чтение состояния БЗ:**
    ```
    02 10 00
    ```
*   **Чтение/установка задержки БЗ:**
    ```
    02 13 00    # Чтение
    02 13 01 XX # Установка, где XX - значение
    ```
*   **Управление аттенюатором корректирующим БЗ:**
    ```
    02 04 00    # Чтение значения
    02 04 01 XX # Установка, где XX - значение в дБ (дискрет 0,5 дБ)
    ```
*   **Управление линией задержки переменной (ЛЗП):**
    ```
    02 10 00    # Чтение значения
    02 10 01 XX # Установка, где XX - значение (0,003-1,75)
    ```

### Команды для Блока БФ (Блок 0x3)
*   **Чтение серийного номера БФ:**
    ```
    03 01 00
    ```
*   **Чтение состояния БФ:**
    ```
    03 10 00
    ```
*   **Чтение/установка модели флюктуаций БФ:**
    ```
    03 15 00    # Чтение
    03 15 01 XX # Установка, где XX - значение
    ```
*   **Управление аттенюатором корректирующим БФ:**
    ```
    03 04 00    # Чтение значения
    03 04 01 XX # Установка, где XX - значение (дискрет 0,5 дБ)
    ```

### Команды получения версий ПО

Возможны запросы версий ПО контроллеров различных блоков. Ответы передаются в бинарном формате:
- Байт 0: ID блока (0x01 для БУ, 0x02 для БЗ, 0x03 для БФ)
- Байт 1: Версия ПО
- Байт 2: День релиза (DD)
- Байт 3: Месяц релиза (MM)
- Байт 4: Год релиза (YY)
- Байты 5-6: CRC16

*   **Запрос версии ПО контроллера БУ:**
    ```
    01 20 00
    ```

*   **Запрос версии ПО контроллера БЗ:**
    ```
    02 20 00
    ```

*   **Запрос версии ПО контроллера БФ:**
    ```
    03 20 00
    ```

*   **Запрос версии ПО платы формирования флуктуаций БФ:**
    ```
    03 30 00
    ```

**Примечание:** Клиент автоматически добавит CRC к этим данным перед отправкой. Все команды должны соответствовать формату: первый байт - идентификатор блока, второй байт - идентификатор устройства, третий байт - код команды (0 - чтение, 1 - запись, 2 - инициализация, 3 - параметры), далее - данные (если требуются).
